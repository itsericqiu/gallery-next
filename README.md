# Gallery Next

Static photography archive for Eric Qiu, built with Astro. This project replaces the older Jekyll gallery and is designed as a quiet, editorial photo archive with selected work, places, thematic series, an archive, and individual photo pages.

Production URL: `https://photos.ericqiu.io`

## Current Shape

- Astro static site deployed to GitHub Pages from `main`.
- Source photos live in git under `src/assets/photos/<Place>/`.
- Photo metadata lives in `src/data/photos.ts`.
- Collection and ordering data lives in `src/data/collections.ts`.
- Astro image optimization generates WebP variants and low-resolution placeholders at build time.
- Photo grids use a small vanilla JS masonry packer for consistent Chrome, Safari, Firefox, desktop, and mobile layout.
- CSS Grid remains the no-JS fallback for photo grids.

## Development

```bash
npm install
npm run dev
```

The local dev server usually runs at `http://localhost:4321/` or `http://127.0.0.1:4321/`.

Before committing code or data changes, run:

```bash
npm run build
```

This runs `astro check` and `astro build`.

## Photo Data Model

Each photo starts as a tuple in `src/data/photos.ts`:

```ts
[id, place, key, width, height, tags]
```

The same file contains editorial metadata keyed by photo id:

```ts
{
  title: 'Short Display Title',
  caption: 'One-sentence editorial caption.',
  alt: 'Concrete accessible description of the image.',
}
```

Important conventions:

- `id` should be stable, usually `<place-slug>-<gallery-number>`, such as `iceland-044`.
- `key` is relative to `src/assets/photos/`, such as `Iceland/eric-qiu-gallery-44.jpg`.
- `width` and `height` must match the source file after resize/copy.
- `alt` is required editorially even though the code has a fallback.
- Tags drive thematic series membership, but series ordering is manually curated in `src/data/collections.ts`.

## Collections

Collections are defined in `src/data/collections.ts`.

- `selected` is the homepage edit. Its `photoIds` order is editorial and should be reviewed manually.
- `archive` uses every photo id from `photos`.
- `place` collections are usually generated by `idsByPlace(...)`.
- `series` collections use explicit arrays in `seriesPhotoIds` so ordering can be curated.

When adding or removing photos, check:

- `selected.photoIds`
- matching `seriesPhotoIds` arrays
- `coverPhotoId` values
- page copy if the collection changes meaningfully

## Adding Photos

Preferred workflow:

```bash
npm run add-photo -- <Place> <file.jpg> [file2.jpg ...]
```

Example:

```bash
npm run add-photo -- Iceland ~/Desktop/eric-qiu-gallery-62.jpg
```

The script:

- resizes each input to 4000px longest edge with macOS `sips`
- copies it to `src/assets/photos/<Place>/`
- prompts for title, caption, alt text, and tags
- inserts the photo into `src/data/photos.ts`
- appends matching series membership in `src/data/collections.ts`
- optionally prepends the photo to the homepage selected collection

After running it, manually review ordering and covers in `src/data/collections.ts`. The script appends series entries; it does not make editorial ordering decisions.

## Image Pipeline

Images are imported from `src/assets/photos` through `src/lib/photo-assets.ts`, then rendered through Astro's `getImage` in `src/lib/media.ts`.

Grid images currently use WebP at bounded widths around `480` and `960`. Detail images use larger bounded widths around `1200` and `2000`. Tiny WebP placeholders are generated for background previews.

There is also a bulk resize helper:

```bash
npm run resize-all
```

Use this carefully. It runs `sips -Z 4000` over every JPG in `src/assets/photos`.

## Masonry Grid

`src/components/PhotoGrid.astro` renders normal photo links first, then a small inline script packs those tiles into columns.

Behavior:

- mobile/tablet at `<=860px`: 2 columns
- editorial desktop grids: 3 columns
- archive desktop grid: 4 columns
- packing uses photo aspect ratios, so it does not wait for image decoding
- tiles preserve original collection order when repacking on resize
- if JS fails or is disabled, CSS Grid fallback still renders usable grids

The `.js` class is added in `src/layouts/BaseLayout.astro` so CSS can hide an unpacked grid briefly and avoid a visible layout jump. Do not remove the fallback grid styles unless replacing the JS behavior with another robust fallback.

## Icons And Favicons

The icon system is SVG-source-first. Source candidates live in:

```text
public/icons/source/
```

Current candidates include city/architecture, architecture-only, landscape, archive-frame, and quiet lowercase `eq` concepts. Preview them locally at:

```text
/icons/preview.html
```

Production icons are generated from one source candidate with:

```bash
npm run icons
```

By default this uses `archive-frame.svg` as the interim production mark while candidates are still being evaluated. To generate from a different candidate:

```bash
npm run icons -- architecture-a
```

Generated outputs include:

- `public/favicon.svg`
- `public/favicon.ico`
- `public/favicon-16x16.png`
- `public/favicon-32x32.png`
- `public/favicon-48x48.png`
- `public/favicon.png`
- `public/apple-touch-icon.png`
- `public/icon-192.png`
- `public/icon-512.png`
- `public/icon-maskable-512.png`
- `public/site.webmanifest`

Do not hand-edit generated PNG/ICO files. Edit or add SVGs in `public/icons/source/`, then rerun `npm run icons`.

The generator applies small output-specific scaling: favicons are enlarged slightly for legibility, app icons use the source proportions, and maskable icons are inset for platform cropping.

## Deployment

GitHub Actions deploys the site to GitHub Pages on pushes to `main`.

Deployment details:

- workflow: `.github/workflows/astro.yml`
- build command in CI: `astro build`
- custom domain file: `public/CNAME`
- Astro site URL: configured in `astro.config.mjs`

The workflow caches Astro's generated image assets with a key based on `src/assets/**`.

## Historical Notes

The project has gone through several layout iterations:

- old Jekyll gallery with static JPG references, previously published at `https://itsericqiu.github.io/gallery/`
- new Astro project with typed photo and collection data
- local git-backed photo assets instead of `public/photos`
- custom-domain deployment at `photos.ericqiu.io`
- CSS column masonry, then CSS Grid for consistency
- final current approach: JS-packed masonry with CSS Grid fallback for cross-browser consistency

The current implementation favors consistent end results across Safari, Chrome, Firefox, desktop, and mobile over relying on browser-specific CSS column balancing.
